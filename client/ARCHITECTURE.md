# Архитектура клиента

## Обзор

Клиент — SPA на **Vue 3** (Composition API, `<script setup>`) с **Vue Router** и **Socket.IO** для real-time коммуникации с сервером. Сборка — **Vite**.

Ключевые задачи клиента:
- Создание и поиск игр через REST + WebSocket-лобби
- Отображение игрового поля с пошаговой механикой
- Синхронизация состояния в реальном времени (ходы, очки, таймер)
- Переподключение к игре при обрыве связи
- Обратная связь пользователю (toast-уведомления, анимации)

## Структура проекта

```
client/src/
├── main.ts              — Bootstrap приложения, глобальный error handler
├── App.vue              — Layout, CSS custom properties, монтирование toast
├── router.ts            — Два маршрута: главная и страница игры
├── config.ts            — API_URL из env-переменной
├── errors.ts            — Маппинг серверных кодов ошибок → локализованные сообщения
├── api/
│   └── games.ts         — REST-вызовы (POST /games)
├── composables/
│   ├── useSocket.ts     — WebSocket-логика игровой сессии
│   ├── useLobby.ts      — WebSocket-логика лобби
│   └── useToast.ts      — Реактивная система уведомлений
├── pages/
│   ├── HomePage.vue     — Создание игры + список доступных игр
│   └── GamePage.vue     — Игровой экран: доска, счёт, таймер, game over
├── components/
│   ├── GameBoard.vue    — CSS Grid обёртка над ячейками
│   ├── GameCell.vue     — Одна ячейка поля с анимацией раскрытия
│   └── ToastNotification.vue — Overlay для toast-уведомлений
└── types/
    └── game.ts          — Типы payload'ов, совпадающие с серверными
```

## Разделение pages/ и components/

- **`pages/`** — роутовые компоненты, привязанные к URL. Содержат бизнес-логику конкретного экрана, вызывают composables и API.
- **`components/`** — переиспользуемые UI-компоненты без привязки к маршрутам. Получают данные через props, отдают события через emit.

Это стандартная конвенция Vue-приложений: страница оркестрирует, компонент отображает.

## Composables

### useSocket

**Роль:** Полный цикл игровой WebSocket-сессии.

Управляет подключением к серверу, подпиской на игровые события и реактивным состоянием партии. Вызывается в `GamePage` при входе на страницу игры.

Реактивное состояние:
| Ref | Назначение |
|-----|-----------|
| `gameState` | Полное состояние партии (`GameStatePayload`) — доска, очки, ход |
| `gameOver` | Payload завершения игры (причина, победитель) |
| `error` | Текст ошибки от сервера (неверный ход, не ваш ход) |
| `unavailable` | Ошибка при попытке войти в игру (не найдена, уже началась) |
| `cancelled` | Флаг: игра отменена создателем |
| `connected` | Статус WebSocket-подключения |
| `playerLeft` | Соперник отключился и пытается переподключиться |
| `extraTurn` | Кратковременный флаг (2 сек): игрок получил бонусный ход |

Методы: `connect()`, `joinGame()`, `openCell()`, `surrender()`, `cancelGame()`, `disconnect()`.

При размонтировании компонента (`onUnmounted`) сокет автоматически отключается.

**Переподключение.** При получении `game:state` токен сохраняется в `sessionStorage`. При повторном входе на страницу игры `joinGame()` проверяет наличие токена и отправляет `game:rejoin` вместо `game:join`. Это позволяет восстановить сессию после перезагрузки страницы или обрыва связи.

### useLobby

**Роль:** WebSocket-подписка на список доступных игр.

Вызывается в `HomePage`. При монтировании подключается к серверу и отправляет `lobby:subscribe`. Сервер push'ит обновлённый список при каждом изменении (создание/удаление/старт игры).

Дополнительно запускает `setInterval` каждые 10 секунд для обновления меток "время назад" без повторного запроса к серверу — просто триггерит реактивность через shallow copy массива.

При размонтировании отписывается и отключает сокет.

### useToast

**Роль:** Глобальная система уведомлений.

Состояние (`toasts`) хранится на уровне модуля (вне функции composable), поэтому все вызовы `useToast()` разделяют один и тот же массив — это singleton по дизайну. Это позволяет вызывать `toast.error(...)` из любого компонента, а `ToastNotification.vue` (смонтированный в `App.vue`) отобразит уведомление.

Каждый toast автоматически удаляется через `duration` миллисекунд (по умолчанию 3000).

```
useToast() в GamePage → push в toasts[] → ToastNotification.vue рендерит
useToast() в HomePage → push в toasts[] ↗
```

## API-слой

### api/games.ts

Содержит REST-вызов `createGame()`. Вынесен из компонента по нескольким причинам:

1. **Компонент отвечает за UI, не за HTTP.** `HomePage` не должен знать, как парсить ответ сервера или извлекать сообщение об ошибке из `Response`.
2. **Переиспользуемость.** Если создание игры понадобится в другом месте, импорт одной функции проще, чем копирование fetch-логики.
3. **Типизация.** `CreateGameParams` и возвращаемый тип `{ gameId: string }` описаны в одном месте.

REST используется только для создания игры — это одноразовый запрос-ответ, не требующий WebSocket. Все остальные взаимодействия идут через сокеты.

### Vite proxy

В dev-режиме запросы на `/games` проксируются на backend:

```typescript
proxy: {
  '/games': {
    target: backendTarget, // из VITE_API_URL
    changeOrigin: true,
  },
}
```

Это позволяет `fetch('/games')` работать без указания полного URL и избежать CORS-проблем при REST-запросах. WebSocket-подключения используют абсолютный URL из `config.ts`.

## Конфигурация

URL бэкенда вынесен в переменную окружения `VITE_API_URL`:

- **`config.ts`** — экспортирует `API_URL`, читая `import.meta.env.VITE_API_URL` с fallback на `http://127.0.0.1:3000`
- **`vite.config.ts`** — использует `loadEnv()` для чтения той же переменной в контексте proxy
- **`env.d.ts`** — типизирует `ImportMetaEnv` для TypeScript автокомплита
- **`.env`** / **`.env.example`** — значение по умолчанию для разработки

Единый источник: все модули импортируют `API_URL` из `config.ts`, а не дублируют строку.

## Обработка ошибок

### Серверные ошибки (WebSocket)

Сервер отправляет ошибки в формате `{ code: string, message?: string }`. Клиент маппит `code` на русскоязычное сообщение в `errors.ts`:

```typescript
export const ERROR_MESSAGES: Record<string, string> = {
  NOT_YOUR_TURN: 'Сейчас не ваш ход',
  CELL_ALREADY_REVEALED: 'Клетка уже открыта',
  // ...
};
```

Fallback-цепочка: локализованное сообщение → `message` от сервера → сам `code`. Это позволяет серверу оставаться language-agnostic.

### Ошибки создания игры (REST)

`api/games.ts` парсит ответ сервера: JSON с полем `message` (может быть массивом от class-validator), или plain text, или fallback с HTTP-статусом. Ошибка выбрасывается как `Error` и ловится в компоненте.

### Глобальный error handler

```typescript
app.config.errorHandler = (err, _instance, info) => {
  console.error(`[Vue Error] ${info}:`, err);
};
```

Перехватывает необработанные исключения в lifecycle hooks и render-функциях, предотвращая "белый экран" без диагностики.

## Компоненты

### GameBoard

CSS Grid обёртка. Принимает двумерный массив `board` и `gridSize`, вычисляет `flatBoard` через `.flat()` для рендера в одномерном `v-for`. Преобразует плоский индекс обратно в `(row, col)` при клике.

Ключ элемента — координаты `${row}-${col}`, а не индекс массива. Это гарантирует корректное переиспользование DOM-элементов при обновлении ячеек.

### GameCell

Отображает одну ячейку поля. Два состояния:

- **hidden** — нераскрытая, может быть `clickable` (ваш ход) или заблокирована
- **revealed** — раскрытая, показывает алмаз или число соседних алмазов

CSS-анимации при раскрытии:
- `cell-reveal` — scale + opacity для обычных ячеек
- `diamond-reveal` — более выразительный scale + rotation для алмазов

Числа подсветки соседних алмазов стилизованы через `data-count` атрибут — каждое значение (0–8) имеет свой цвет, аналогично классическому Minesweeper.

### ToastNotification

Рендерится через `<Teleport to="body">` — вне DOM-дерева приложения, чтобы не зависеть от `overflow` или `z-index` родителей. `<TransitionGroup>` обеспечивает анимацию появления/исчезновения.

## CSS-архитектура

### Custom properties

Все цвета и радиусы определены в `:root` в `App.vue` и используются во всех компонентах через `var(--color-*)`. Это обеспечивает:

- **Консистентность** — невозможно случайно использовать "почти тот" оттенок
- **Изменяемость** — смена палитры в одном месте затрагивает всё приложение
- **Читаемость** — `var(--color-accent)` понятнее, чем `#e94560`

Переменные организованы по группам: фоны, акценты, семантические цвета (info/warning/error/success), текст, игроки, ячейки, радиусы.

### Scoped styles

Все компоненты используют `<style scoped>`. Глобальные стили ограничены `App.vue` — только reset, `:root` переменные и layout приложения.

## Типы

`types/game.ts` содержит клиентские версии серверных payload'ов:

| Тип | Когда используется |
|-----|--------------------|
| `ClientCell` | Ячейка доски (скрытые поля не раскрыты) |
| `GameStatePayload` | Полное состояние при `game:state` |
| `CellRevealedPayload` | Обновление одной ячейки при `game:update` |
| `GameOverPayload` | Результат партии при `game:over` |
| `AvailableGameInfo` | Элемент списка лобби |

Типы дублируют серверные интерфейсы намеренно — клиент и сервер не разделяют пакеты, и клиентские типы содержат только то, что приходит по сети (без серверных `Map`, таймеров и т.д.).

## Поток данных на странице игры

```
GamePage (mount)
  ├── useSocket().connect()       → WebSocket подключение
  ├── watch(connected) → joinGame() → emit('game:join' / 'game:rejoin')
  │
  ├── on('game:state')            → gameState ref обновляется
  │   └── GameBoard + GameCell рендерятся реактивно
  │
  ├── on('game:update')           → одна ячейка в board обновляется
  │   ├── GameCell анимирует раскрытие
  │   └── extraTurn ref на 2 сек → баннер "Бонусный ход!"
  │
  ├── on('game:over')             → gameOver ref → overlay с результатом
  │
  ├── on('game:error')            → error ref → показ в UI
  │
  ├── on('game:unavailable')      → toast.error() + redirect на главную
  │
  ├── on('game:player-left')      → playerLeft ref → предупреждение
  │
  └── computed(timeLeft)          → обратный отсчёт из turnStartedAt
      └── now ref обновляется каждые 250мс через setInterval
```

## Поток данных на главной странице

```
HomePage (mount)
  ├── useLobby()
  │   ├── io(API_URL) → emit('lobby:subscribe')
  │   ├── on('lobby:list') → games ref (реактивный список)
  │   └── setInterval → shallow copy для обновления "X мин. назад"
  │
  └── onCreateGame()
      ├── Валидация (нечётное, < N²)
      ├── createGame(params)      → POST /games
      └── router.push(`/game/${gameId}`)
```
