## Почему проект сделан именно так

### 1. Сервер специально декомпозирован на несколько сервисов

Я сознательно разделил серверную логику на несколько сервисов:

- `LobbyService` отвечает за создание игры, список доступных игр, вход в лобби и отмену игры
- `GameplayService` отвечает за правила игры: открытие клеток, подсчёт очков, завершение партии, таймер хода
- `SessionService` отвечает за переподключение игрока и обработку дисконнектов
- `GameRepositoryService` отвечает за хранение состояния игр
- `BoardGeneratorService` отвечает только за генерацию поля

С точки зрения **SOLID** такое разделение правильнее, потому что у каждого сервиса одна зона ответственности. Код проще читать, тестировать и расширять.

### 2. Стейт игры вынесен в отдельный репозиторий и сделан абстрактным

Сейчас состояние игр хранится in-memory через `Map`, что достаточно для тестового задания. При этом репозиторий сделан абстрактным:

- текущая реализация простая и быстрая
- при необходимости её можно заменить без переписывания бизнес-логики
- в теории вместо in-memory репозитория можно будет подключить **Redis** или другую внешнюю систему хранения

То же самое относится и к генератору поля: он вынесен отдельно, чтобы игровая логика не зависела от конкретного алгоритма генерации.

### 3. Добавлен таймер хода

Я специально добавил ограничение по времени на ход, чтобы игроки не пытались выиграть за счёт искусственного затягивания партии.

Это частая проблема в пошаговых играх. Самый понятный аналог — **шахматы**, где время является частью правил и защищает игру от злоупотреблений.

В результате игровой процесс становится:

- более честным
- более динамичным
- более предсказуемым с точки зрения UX

### 4. Состояние игры транслируется по сокету каждому игроку

Основная логика построена вокруг **Socket.IO**, потому что игра real-time и состояние должно синхронно обновляться у обоих игроков:

- подключение к игре
- открытие клетки
- смена хода
- обновление счёта
- завершение игры
- отключение и переподключение игрока

После каждого важного действия сервер отправляет актуальное состояние или событие обоим участникам. Это даёт живую синхронизацию без постоянного polling.

### 5. Сервер не раскрывает лишнюю информацию клиенту

В клиент отправляется не полное внутреннее состояние клетки, а его безопасная версия. Пока клетка не раскрыта, клиент не знает, есть ли там алмаз.

Это сделано специально, чтобы нельзя было "выиграть" просто посмотрев payload в DevTools.

### 6. Ошибки передаются кодами, а не готовыми текстами

Сервер отправляет клиенту коды ошибок, а клиент уже маппит их на пользовательские сообщения.

Это решение даёт несколько плюсов:

- сервер остаётся независимым от языка интерфейса
- локализацию проще менять на клиенте
- сообщения для пользователя и техническая логика не смешиваются

### 7. Используются типовые алиасы и indexed access types

В коде местами используются не просто `string` и `number`, а конструкции вроде:

- `Game['id']`
- `Game['gridSize']`
- `CellRevealedPayload['row']`

Это сделано для того, чтобы типы были связаны между собой через единый источник правды. Если меняется тип в основной модели, все зависимые сигнатуры автоматически подхватывают это изменение.

Это немного увеличивает формальность кода, но делает его более устойчивым к дальнейшим изменениям.

### 8. Клиент тоже разделён по ответственности

На клиенте логика разделена на:

- `pages/` — роутовые страницы
- `components/` — UI-компоненты
- `composables/` — переиспользуемая реактивная логика
- `api/` — REST-запросы

Такое разделение помогает не смешивать UI, сетевую логику и состояние.

## Правила игры

- Играют два игрока
- Поле имеет размер `N x N`
- На поле размещено `M` алмазов
- `M` делается нечётным, чтобы в обычном сценарии был гарантирован победитель
- Если игрок открывает алмаз, он получает очко и дополнительный ход
- Если игрок открывает число, ход переходит сопернику
- Игра заканчивается, когда все алмазы найдены или когда кто-то проигрывает по таймеру / сдаётся

## Технологии

- **Server:** NestJS, Socket.IO, TypeScript, class-validator, ESLint, Prettier
- **Client:** Vue 3, Vite, Vue Router, Socket.IO Client, TypeScript

## Запуск проекта

### Требования

- Node.js >= 18

### Server

```bash
cd server
npm install
npm run start:dev
```

Сервер будет доступен на `http://localhost:3000`.

### Client

```bash
cd client
npm install
npm run dev
```

Клиент будет доступен на `http://localhost:5173`.

## Как поиграть

1. Открыть `http://localhost:5173`
2. Создать новую игру
3. Скопировать `gameId`
4. Передать его второму игроку
5. Второй игрок открывает приложение и подключается к игре
6. После подключения второго игрока партия стартует автоматически

## API

### REST

- `GET /games` — получить список доступных игр
- `POST /games` — создать новую игру

Пример тела запроса:

```json
{
  "gridSize": 5,
  "diamondsCount": 5,
  "turnTimeSeconds": 30
}
```

Пример ответа:

```json
{
  "gameId": "uuid"
}
```

### WebSocket события

**Client → Server**

- `lobby:subscribe`
- `lobby:unsubscribe`
- `game:join`
- `game:rejoin`
- `game:cancel`
- `game:open`
- `game:surrender`

**Server → Client**

- `lobby:list`
- `game:state`
- `game:update`
- `game:over`
- `game:error`
- `game:unavailable`
- `game:cancelled`
- `game:player-joined`
- `game:player-reconnected`
- `game:player-left`

## Архитектурные заметки

Я понимаю, что для тестового задания такую декомпозицию можно посчитать чуть более сложной, чем минимально необходимый MVP. Это осознанный выбор.

Мне было важно показать не только рабочий результат, но и то, как я думаю об архитектуре:

- как разделяю ответственность
- как проектирую код на расширение
- как изолирую бизнес-логику от транспорта
- как не даю клиенту лишние данные
- как делаю real-time механику устойчивой к дисконнектам
